# Firebase Clean-Slate Migration — Risky Assumptions

**Feature:** Move Glidey client from legacy Firebase project (`auth-bf4f5`) to a new GCP project's Firebase.
**Scope:** Firebase Auth setup, mobile config swap, backend service account + env var update, auth providers, `ensureClientProfile` bootstrap.
**Date:** 2026-04-21

---

## Summary

The migration is primarily a configuration swap (new `google-services.json`, `GoogleService-Info.plist`, and `config/firebase.ts`), but several structural risks lurk underneath. The codebase still initialises Firestore (`getFirestore`) and Cloud Functions (`getFunctions`) inside `packages/auth/src/firebase.ts`, even though the backend has moved to a PostgreSQL monolith at `34.140.138.4`. Phone auth in `usePhoneAuth.ts` falls through to a `sendPhoneOTPPublic` / `verifyPhoneOTPPublic` Cloud Functions call path, which will be broken the moment the old project is abandoned unless those functions are also deployed to the new project. The iOS `GoogleService-Info.plist` does not yet exist in the repo (glob returned no results), meaning the iOS build will break until it is added to `app.config.js` and the native layer. These concrete gaps inform the assumptions below.
