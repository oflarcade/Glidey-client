# Booking Module — PM Assumption Analysis

Feature: Connecting BookingSheet UI to the real backend booking API (`POST /rides`, `PATCH /rides/{id}/status`, `GET /rides/history`).

## Assumption Register

| # | Category | Assumption | Confidence | Evidence / Risk Signal | Fast Test / Experiment |
|---|----------|-----------|------------|------------------------|------------------------|
| A1 | **Feasibility** | The UI's `fareEstimates: FareEstimateItem[]` prop (which drives `ScooterCarousel` and gates `canBook`) can be replaced with fare fields returned by `POST /rides` itself (`fareBase`, `farePerKm`, `farePerMin`), allowing the user to see a price before the ride is live. | **LOW** | `POST /rides` creates the ride immediately in `pending` status — there is no dry-run / estimate mode. The carousel was designed for a pre-booking estimate step that does not exist. Showing fare requires creating a real ride first, which means either (a) deleting it on cancel or (b) fundamentally rethinking the UX. | Audit the backend: call `POST /rides` then immediately `PATCH /rides/{id}/status` `{ status: "cancelled" }` and confirm no billing or driver disturbance occurs. If safe, fare-on-create is viable; if not, a static formula must be derived from `fareBase`/`farePerKm`/`farePerMin` and displayed as an estimate before booking. |
| A2 | **Feasibility** | `GET /rides/history` (polling) is a viable substitute for a missing `GET /rides/{id}` endpoint during the `searching → matched` flow. | **MEDIUM** | The history list is paginated; polling the full list to detect a single status change is expensive and introduces latency. The polling interval, page size, and whether the latest ride appears at the top of page 1 are undocumented. `useMatching` hook currently calls `rideId`-scoped polling that assumes a direct-fetch endpoint. | Measure: poll `GET /rides/history?limit=1` every 3 s, log time-to-detection for status changes in a staging environment. Confirm latest ride is always page-1 element. If >10 s p50 latency → escalate SSE/WebSocket investigation before shipping. |
| A3 | **Feasibility** | Undocumented SSE/WebSocket realtime channel exists and can replace history polling for matching status. | **LOW** | Mentioned in problem statement as "likely exists but not documented." No endpoint, auth schema, or event payload is confirmed. The UI has a `RetryTimeline` (3-attempt model) that implies a push-or-poll loop. | Inspect VM network config: `ss -tlnp` on `34.140.138.4`, check for long-lived connection endpoints. Review backend source at `/opt/EMIN/glidey/RentAScooter-Backend-Monolith/` for Socket.io / SSE emitters. Document findings before building the polling fallback path. |
| A4 | **Value** | Riders will book without seeing a firm total fare upfront — accepting `fareBase + farePerKm + farePerMin` rate card instead of a pre-computed total. | **MEDIUM** | West African mobility apps (InDriver, Yango) typically show a price before the ride is confirmed. Glidey's backend only returns `fareTotal` on completion. The `formatXOF` helper in BookingSheet already formats numbers, so a computed estimate is technically easy, but riders may distrust a formula they must calculate themselves. | Run a 5-person contextual inquiry in Dakar: show two prototype screens — (a) "Tarif estimé: 2 400 XOF" vs (b) "Base 500 + 120 XOF/km · 3.2 km". Measure comprehension and booking intent. |
| A5 | **Usability** | Removing the `ScooterCarousel` (vehicle-type selector) does not degrade booking intent — riders are indifferent to vehicle type at booking time. | **LOW** | The carousel is the visual centrepiece of `BookingModeContent`. Without a `/vehicle-types` catalog, it can only show real options if populated from nearby-driver profiles, which adds a dependency on `GET /drivers/nearby` results. An empty carousel currently blocks `canBook` (`selectedEstimate !== null` gate on line 379). If removed without a replacement selection mechanism, the CTA is always unblocked, which changes trust dynamics. | A/B test (Maze or Figma prototype): group A sees the carousel, group B sees a single "Réserver maintenant" CTA with fare rate card. Measure drop-off rate and post-booking satisfaction. |
| A6 | **Value** | Cash-only payment ("Espèces", hardcoded line 130) is acceptable to the target rider segment for the initial launch. | **HIGH** | Hardcoded in UI, no payment method selector, and the backend API shows no payment object on ride creation. This is likely an intentional constraint for Phase 4. Risk is low for feasibility but high for growth ceiling — mobile money (Wave, Orange Money) dominates Senegal. | Validate with 10 rider interviews: "Would you use Glidey if only cash was accepted?" and "What payment method do you use most for transport?" Use responses to backlog mobile-money as a Phase 5 requirement. |
| A7 | **Viability** | A 3-attempt retry model (hardcoded in `RetryTimeline` / `searchingSubtitle`) is sufficient to find a driver before rider abandonment. | **MEDIUM** | The retry loop is a UI concept — there is no backend retry orchestration visible in the API spec. The actual matching is likely demand-driven (nearest available driver accepts). The "3 attempts" framing may set a false expectation if the backend does not implement attempt boundaries. | Instrument: log `POST /rides` timestamp and first `status !== "pending"` detection timestamp from history polling. Compute p50/p95 wait time over 50 test rides. If p95 > 90 s, the 3-attempt copy is misleading and the fallback ("Aucun conducteur disponible") will trigger too often. |
| A8 | **Feasibility** | `vehicleType` as a free-form string on driver profile is sufficient to route a ride to the correct vehicle — no structured vehicle catalog is needed in the booking payload. | **MEDIUM** | `POST /rides` body has no `vehicleType` field, so the rider cannot request a specific type. Matching is purely proximity-based. This contradicts the carousel UX, which implies the rider's selection influences which driver is dispatched. If drivers have mixed fleet (moto / jakarta / voiture), the rider may get a vehicle type they did not expect. | Review backend matching logic in `/opt/EMIN/glidey/RentAScooter-Backend-Monolith/` — confirm whether driver dispatch is FIFO/nearest or filtered by type. If unfiltered, remove the carousel and update copy to "Véhicule disponible le plus proche." |
| A9 | **Ethics / Trust** | Riders will trust a fare computed client-side from `fareBase + (farePerKm × distanceM/1000) + (farePerMin × durationS/60)` using the directions API values. | **MEDIUM** | Route distance from `GET /directions` is used as the billing proxy, but actual driven distance may differ (detours, traffic). If the final `fareTotal` (returned only at completion) is materially higher than the pre-ride estimate, riders will feel deceived. This is a retention and regulatory risk in a market with price-sensitive users. | Prototype the estimate formula. Run 20 paired rides: record estimated vs actual fareTotal. If variance >15% on >20% of rides, implement a disclosure banner ("Tarif estimé — le prix final peut varier") and surface the rate card explicitly. |
| A10 | **Go-to-Market** | French-only UI (`"Réserver maintenant"`, `"Calcul du tarif…"`, `"Recherche d'un conducteur…"`) is sufficient for the Dakar launch — no Wolof localisation is needed. | **MEDIUM** | The `@rentascooter/i18n` package supports `fr`/`en` only. Dakar's urban population is French-literate, but a significant share of riders in peri-urban zones (Guédiawaye, Pikine) prefer Wolof. Competitor apps (Yassir) are French-only, so parity is acceptable short-term. | Track language setting in analytics at launch. If >15% of users switch to English (proxy for non-French preference), initiate Wolof localisation spike. |
| A11 | **Strategy** | The `PATCH /rides/{id}/status { status: "cancelled" }` endpoint can be safely called by the rider at any state (pending, accepted, active) without penalty or backend error. | **LOW** | Cancel behaviour at each state transition is undocumented. Cancelling an `active` ride (driver en route or ride started) likely has different semantics than cancelling `pending`. The UI exposes a single "Annuler la recherche" button across all non-matched states — no guard distinguishes late cancellation from pre-match cancellation. | Test all cancel paths against staging: cancel at `pending`, `accepted`, and `active`. Confirm HTTP response codes, any driver notification, and whether `fareTotal` is computed on active cancels. Build a cancellation policy matrix before the cancel button goes live. |
| A12 | **Team** | The team can deliver polling-based matching (history page-1 status watch) as a stable temporary solution while SSE/WebSocket is investigated — without needing a backend change. | **MEDIUM** | `GET /rides/history` is a paginated list endpoint, not a streaming endpoint. Polling at 3–5 s intervals is workable but will require careful error handling (network loss, auth token expiry mid-poll, app backgrounding on iOS). The mobile team has no current hook for background polling lifecycle. | Spike: implement `useRideStatusPolling(rideId)` using React Query `refetchInterval`, test on physical iOS device with app backgrounded for 30 s and with network toggle. Validate that status change is detected within 10 s of backend update before marking the approach as production-ready. |

## Priority Order for Testing

1. **A1** — fare display without a dry-run endpoint (blocks entire booking flow design)
2. **A11** — cancel safety across states (safety-critical, must not create billing or driver disputes)
3. **A3 / A2** — realtime vs polling (determines architecture of the matching screen)
4. **A8** — vehicle type routing (determines whether carousel stays or is cut)
5. **A4 / A9** — fare trust and transparency (retention risk)
6. **A5** — carousel removal impact (UX risk, lower stakes)
7. **A7** — 3-attempt framing (copy/comms risk)
8. **A12** — polling feasibility (team capacity risk)
9. **A6** — cash-only (known constraint, low immediate risk)
10. **A10** — language coverage (growth risk, post-launch)
