# Cavekit: Location Search

## Scope
Everything the rider app does to let a user pick a destination: typing a query, seeing suggestions, converting a selected suggestion into real coordinates, and persisting confirmed destinations into a per-user history that is surfaced on later sessions. This kit replaces the previous Mapbox-Search-Box-plus-Firebase-callable flow with a Google-Places-backed flow exposed by the new REST backend. It also covers the shared `Suggestion` type migration that forces downstream components (hooks, modals) to drop Mapbox-specific fields.

## Requirements

### R1: Address autocomplete
**Description:** Given a query string of at least two characters, the system returns a list of place suggestions. Each suggestion exposes enough information for the UI to render it and for a downstream detail resolution step (see R2) to turn it into map coordinates. No session token is generated, tracked, or passed by the client.

**Acceptance Criteria:**
- [ ] A query of length `< 2` characters returns an empty list without triggering a backend call.
- [ ] A query of length `>= 2` characters returns a list of suggestion objects (possibly empty when nothing matches).
- [ ] Each returned suggestion exposes a stable `placeId`, a `name` (display label), and a `formattedAddress`.
- [ ] No session-token value is generated by the client, passed in the request, or stored in client state.
- [ ] The call is authenticated via the shared API client (see `cavekit-api-client.md` R1).

**Dependencies:** `cavekit-api-client.md` R1, R3.

---

### R2: Place detail resolution
**Description:** Given a `placeId` returned from autocomplete, the system returns the full resolved location for that place: latitude, longitude, a display name, and a formatted address. This is the canonical way to turn a user's suggestion pick into coordinates the map can use.

**Acceptance Criteria:**
- [ ] Calling detail resolution with a valid `placeId` returns a location object containing numeric `latitude`, numeric `longitude`, a `name`, and an `address`.
- [ ] Calling detail resolution with an unknown or invalid `placeId` surfaces a typed error to the caller.
- [ ] The call is authenticated via the shared API client.
- [ ] A suggestion chosen from the autocomplete list can be converted into coordinates exclusively through this mechanism (no coordinates are ever expected on the autocomplete suggestion itself).

**Dependencies:** R1, `cavekit-api-client.md` R1, R3.

---

### R3: Location history retrieval
**Description:** The authenticated user's previously selected destinations can be retrieved as a list. Each entry exposes enough information to display it as a recent destination and to select it directly without re-running autocomplete.

**Acceptance Criteria:**
- [ ] A location history fetch for a signed-in user returns a list of history entries (possibly empty).
- [ ] Each entry exposes `address`, `name`, `latitude`, `longitude`, a `frequency` count, and a `lastUsedAt` timestamp.
- [ ] The call is authenticated via the shared API client.
- [ ] A user with no prior confirmed destinations receives an empty list, not a typed error.

**Dependencies:** `cavekit-api-client.md` R1, R3.

---

### R4: Location history persistence
**Description:** When the user confirms a destination (i.e. commits to it as the drop-off for the trip), that destination is persisted to the user's location history so that it can appear on future retrievals (see R3).

**Acceptance Criteria:**
- [ ] Confirming a destination triggers a persistence call carrying the confirmed address, name, latitude, and longitude.
- [ ] A subsequent R3 retrieval on the same user account includes the newly persisted destination.
- [ ] Repeated confirmations of the same destination update the `frequency` count and `lastUsedAt` rather than creating a duplicate entry (the client relies on the backend's idempotency; it does not deduplicate locally).
- [ ] Persistence failures do not block the user's trip flow (they are surfaced as a typed error but the user's destination selection still proceeds in the UI).

**Dependencies:** R3, `cavekit-api-client.md` R1, R3.

---

### R5: Updated `Suggestion` type
**Description:** The shared TypeScript `Suggestion` type used across hooks, components, and services must reflect the new place-identifier model. It must no longer expose Mapbox-specific fields and must no longer include a session-token field.

**Acceptance Criteria:**
- [ ] The `Suggestion` type exposes `placeId` as its stable identifier field.
- [ ] The `Suggestion` type does not contain a field named `mapboxId`.
- [ ] The `Suggestion` type does not contain any of: `maki`, `poiCategory`, `featureType`, `context`.
- [ ] The `Suggestion` type does not contain a `sessionToken` field.
- [ ] The project compiles in strict TypeScript mode with the new type; no downstream consumer of `Suggestion` references a removed field.

**Dependencies:** R1.

---

### R6: Hook compatibility (autocomplete flow)
**Description:** The autocomplete hook consumed by the destination UI must operate without generating or managing a session token. The end-to-end flow stays: the user types, the hook returns suggestions carrying `placeId`, the user picks one, and detail resolution converts that `placeId` into coordinates.

**Acceptance Criteria:**
- [ ] The autocomplete hook does not create, store, or pass a session token.
- [ ] When the query changes, the hook surfaces suggestions matching the R1 shape.
- [ ] When the user selects a suggestion, the hook (or its caller) uses R2 to resolve coordinates from the suggestion's `placeId`.
- [ ] The hook's public surface compiles without reference to removed Mapbox fields (R5).

**Dependencies:** R1, R2, R5.

---

### R7: Search-mode content surface in the unified booking sheet
**Description:** The destination-selection UI is no longer a standalone overlay. It is the **search-mode content surface** of the unified booking sheet (see `cavekit-booking-sheet-ux.md` R10). It renders the autocomplete input, the list of suggestions, and the location-history list; selecting a suggestion triggers detail resolution (R2) and transitions the unified sheet to booking mode. The search input receives keyboard focus automatically whenever search mode is entered. The search surface cannot be opened or dismissed independently of the unified sheet's mode machine.

**Acceptance Criteria:**
- [ ] A full-codebase search finds zero instances of a standalone destination-search modal/overlay mounted in the application's component tree outside of the unified booking sheet's search-mode content.
- [ ] The search-mode content surface renders autocomplete suggestions using only the R5 `Suggestion` shape.
- [ ] The search-mode content surface renders location-history entries (R3) alongside autocomplete results, using the same `Suggestion`-compatible rendering path as autocomplete results.
- [ ] Selecting a suggestion from the search-mode surface triggers a detail-resolution call (R2) before the destination is handed to the trip flow.
- [ ] Selecting a suggestion from the search-mode surface transitions the unified booking sheet to booking mode (see `cavekit-booking-sheet-ux.md` R10).
- [ ] The search input receives keyboard focus automatically within one animation frame of the unified sheet entering search mode; no additional user tap is required to focus the input.
- [ ] The search-mode content surface holds no state, prop, or ref named `sessionToken` (or any equivalent).
- [ ] There is no code path in the application that opens or dismisses the destination-search UI independently of the unified booking sheet's sheet-mode atom.

**Dependencies:** R2, R3, R5, R6, `cavekit-booking-sheet-ux.md` R10.

---

### R8: Demo mode
**Description:** When the app runs in demo mode, autocomplete, place detail, and location history all return hardcoded Dakar-area mock data without hitting the backend. This lets the UI be exercised end-to-end on a device that has no network or no backend access.

**Acceptance Criteria:**
- [ ] In demo mode, autocomplete for any query of length `>= 2` returns a non-empty list of mock suggestions with `placeId`, `name`, and `formattedAddress` fields populated.
- [ ] In demo mode, detail resolution of any mock `placeId` returns a Dakar-area location with real numeric `latitude` and `longitude`.
- [ ] In demo mode, location history returns a non-empty list of Dakar-area mock entries with all fields from R3 populated.
- [ ] In demo mode, no outgoing HTTP request is dispatched by any of the three operations.
- [ ] Mock data satisfies the same TypeScript types as real data; no `any` escapes.

**Dependencies:** R1, R2, R3, R5, `cavekit-api-client.md` R4.

## Out of Scope
- WebSocket or streaming updates to suggestions or history.
- Any continued use of Mapbox Search Box or a `mapboxId` field (fully removed).
- Session-token billing semantics or any client-managed session lifecycle.
- Reverse geocoding from raw coordinates back into a formatted address.
- Fuzzy/typo correction beyond whatever the backend provides.
- Favorites or named saved places (history is the only persisted collection).
- Standalone modal overlay — the search UI is a mode of the unified booking sheet, not a freestanding component (see `cavekit-booking-sheet-ux.md` R10).
